SCA Academia is a full-stack MERN application split into two independent processes: a Node.js/Express REST API backed by MongoDB, and a React 18/Vite single-page application that communicates with the API exclusively over HTTP. The two sides share no code at runtime — the frontend consumes the API through TanStack Query, which handles caching, background refetching, and optimistic updates.

## MERN stack overview

## Backend structure

The `backend/` directory follows an MVC-adjacent layout with routes, controllers, models, and middleware as separate concerns.

```
backend/
├── server.js              # Entry point: connects DB, registers middleware + routes
├── seed.js                # Seeds core user accounts
├── models/                # Mongoose schemas (Student, Teacher, User, Finance, …)
├── routes/                # Express routers — one file per domain
├── controllers/           # Business logic called by routers
├── middleware/
│   └── authMiddleware.js  # JWT verification and RBAC (protect, restrictTo)
├── uploads/               # Static files served at /uploads
└── logs/                  # PM2 log output (production)
```

### Middleware order

The order of middleware registration in `server.js` is intentional and must not be changed:

1. `cors()` — must run before any route or body parser
2. `express.json({ limit: "50mb" })` — parses JSON bodies; 50 MB limit supports Base64-encoded images
3. `cookieParser()` — must run before auth middleware reads `req.cookies`
4. `express.urlencoded()` — handles form-encoded bodies
5. Static file serving at `/uploads`
6. Debug request logger
7. API route handlers

## Authentication and security

Authentication is JWT-based with tokens stored exclusively in **HTTP-only cookies** — they are never written to `localStorage` or exposed to JavaScript. The `cookie-parser` middleware makes the token available to `authMiddleware.js` via `req.cookies.authToken`.

The middleware actively **rejects** any request that sends a token in the request body, returning `403 Forbidden` with a security violation message. This prevents accidental or malicious token leakage through API clients that serialize credentials into request bodies.

```javascript
// middleware/authMiddleware.js — protect pattern
const protect = async (req, res, next) => {
  const token = req.cookies.authToken; // cookies only — never req.body
  if (!token) return res.status(401).json({ message: "Authentication required." });
  const decoded = jwt.verify(token, process.env.JWT_SECRET);
  req.user = await User.findById(decoded.id);
  next();
};

const restrictTo = (...roles) => (req, res, next) => {
  if (!roles.includes(req.user.role))
    return res.status(403).json({ message: "Insufficient permissions." });
  next();
};
```

<Accordion title="Role hierarchy — OWNER, PARTNER, STAFF">

  **OWNER**
  - Full system access across all modules
  - Can create, activate, and deactivate STAFF accounts
  - Manages expense distribution to partners
  - Accesses all financial data including aggregate partner views
  - Core account: `canBeDeleted: false` — cannot be removed via the API

  **PARTNER**
  - Views only their own revenue stream and wallet balance
  - Can perform the **daily closing** operation that moves `floatingCash` into confirmed `walletBalance`
  - Sees their own `pendingDebt` (owed to the OWNER for shared expenses)
  - Cannot view other partners' financial data
  - Core account: `canBeDeleted: false`

  **STAFF / OPERATOR**
  - Operational access: student enrollment, fee collection, attendance, gatekeeper
  - No financial visibility — payroll, expense, and partner data is hidden
  - Accounts are created and managed exclusively by the OWNER
  - Can be activated or deactivated via `PATCH /api/auth/staff/:id/toggle`

  Each `User` document also carries financial fields used during daily operations:
  - `walletBalance` — confirmed, closed cash
  - `floatingCash` — collected during the day but not yet "closed"
  - `pendingDebt` — amount owed back to the OWNER for distributed expenses

</Accordion>

## Data flow

A typical read/write cycle from the browser to the database follows this path:

```
React component
  → TanStack Query (useQuery / useMutation)
    → fetch() to REST API (VITE_API_URL/api/*)
      → Express router (routes/*.js)
        → Controller function (controllers/*.js)
          → Mongoose model (models/*.js)
            → MongoDB document store
```

TanStack Query manages the client-side cache. After a successful mutation (creating a student, recording a payment), the relevant query keys are invalidated so the UI refetches fresh data automatically without a page reload.

## Transaction and fee lifecycles

Two domain-specific state machines drive the core financial logic:

**Transaction lifecycle (finance module)**

```
FLOATING  →  VERIFIED
```

Transactions start as `FLOATING` when a fee is collected during the day. The PARTNER or OWNER runs a **daily closing** operation which confirms all floating transactions for that session, moving them to `VERIFIED` and adding the totals to `walletBalance`. This separates in-progress collections from confirmed revenue.

**Student fee status lifecycle**

```
pending  →  partial  →  paid
```

When a student's `paidAmount` is zero the fee status is `pending`. Any partial payment moves it to `partial`. When `paidAmount` equals `totalFee` the status becomes `paid`. The `balance` field is a Mongoose virtual (`totalFee - paidAmount`) and is never persisted.
