title: "Notifications API — In-App Alerts for Finance and System Events"
description: "Fetch, mark as read, and delete in-app notifications targeted at the current user by recipient ID or role. Notifications are generated automatically by finance, payroll, and gatekeeper operations."
---

The Notifications API surfaces in-app alerts generated by other parts of the system. Notifications are created automatically — for example, when a teacher payout is processed, when a fee is collected, or when a payout request is submitted. The API exposes read and delete operations so the frontend can manage notification state.

Notifications are targeted either at a specific user (`recipient`) or at everyone with a given role (`recipientRole`). The `GET /api/notifications` endpoint returns items that match either condition for the currently authenticated user.

## Endpoints

### Get notifications for the current user

```
GET /api/notifications
```

Returns the 10 most recent notifications for the authenticated user, ordered newest-first. Includes notifications addressed to the user directly (`recipient`) and notifications addressed to their role (`recipientRole`).

**Response**

<ResponseField name="success" type="boolean" />
<ResponseField name="data" type="Notification[]">
  Array of notification objects, newest first. See [Notification fields](#notification-fields).
</ResponseField>
<ResponseField name="unreadCount" type="number">
  Number of notifications in the response where `isRead` is `false`.
</ResponseField>

---

### Mark notifications as read

```
POST /api/notifications/mark-read
```

Marks one or all notifications as read for the current user.

**Request body**

<ParamField body="notificationId" type="string">
  MongoDB ObjectId of a single notification to mark as read. Ignored when `markAll` is `true`.
</ParamField>

<ParamField body="markAll" type="boolean">
  When `true`, marks every unread notification for the current user (by recipient ID or role) as read.
</ParamField>

<Note>
  At least one of `notificationId` or `markAll: true` must be provided, otherwise the request succeeds but makes no changes.
</Note>

**Response**

<ResponseField name="success" type="boolean" />
<ResponseField name="message" type="string">`"Notifications marked as read"`</ResponseField>

## Notification fields

<ResponseField name="_id" type="string">MongoDB ObjectId.</ResponseField>
<ResponseField name="recipient" type="string">
  ObjectId of the target User. May be `null` for role-targeted notifications.
</ResponseField>
<ResponseField name="recipientRole" type="string">
  Target role. One of `OWNER`, `PARTNER`, or `STAFF`. May be `null` for user-specific notifications.
</ResponseField>
<ResponseField name="message" type="string">
  Human-readable notification message, e.g. `"PKR 15,000 paid to Ahmed Khan by Admin"`.
</ResponseField>
<ResponseField name="type" type="string">
  Notification category: `FINANCE` (triggered by fee collection and payouts) or `SYSTEM` (default, for general alerts).
</ResponseField>
<ResponseField name="isRead" type="boolean">
  `false` until explicitly marked read via the `mark-read` endpoint.
</ResponseField>
<ResponseField name="relatedId" type="string">
  String reference to the related document — typically a Transaction ObjectId, a payout request ObjectId, or a voucher ID string.
</ResponseField>
<ResponseField name="createdAt" type="string">ISO 8601 creation timestamp.</ResponseField>
<ResponseField name="updatedAt" type="string">ISO 8601 last-updated timestamp.</ResponseField>

## When notifications are created

Notifications are created automatically by other API operations. The table below summarises the main triggers.

| Trigger | `type` | Targeted at |
|---|---|---|
| Student fee collected (misc payment) | `FINANCE` | `OWNER` role |
| Teacher payout processed via wallet debit | `FINANCE` | `OWNER` role |
| Teacher payout request submitted | `FINANCE` | All `OWNER` users |
| Payout request approved | `FINANCE` | Teacher (recipient) |
| Payout request rejected | `FINANCE` | Teacher (recipient) |
| Manual payroll credit added | `FINANCE` | `OWNER` role |
